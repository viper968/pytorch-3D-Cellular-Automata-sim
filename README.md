# pytorch-3D-Cellular-Automata-sim
a pytorch and numpy based 3D cellular automata simulator with heavy regard for optimization in processing power, RAM/VRAM usage, and storage used 

**NOTE**: this is a random project that might get updated occasionally if i decide to add more things to it or if it needs to be updated (python libs removing/adding/changing features, blender python changing)

current features:
- processing via pytorch bf16 tensors to allow VERY fast processing of massive grids
- standard s/b/c/n formatting for CA rules along with a few presets
- asymetric sized supported, meaning grid sizes of `400*50*400` are possible
- 4 starting grid modes supported:
  1. soup, random fill of entire grid with density controlled by PROB_ALIVE
  2. center_block, random fill of a smaller block in center of the grid, block size controlled by SEED_BLOCK_SIZE and density controlled by PROB_ALIVE
  3. single, a single cell activated in the center of the grid
  4. shell, similar to center_block but is only 1 cell thick with the center hollowed out, shell density and size controlled the same way as center_block
- saves simulations as .npz (zipped numpy data format) with a slightly custom zip file packer to allow multithreaded compression due to numpy not allowing multithreadded compression of normal .npz files
- data stored in a delta format to majorly reduce space used per frame
- an overengineered unit8+uint16 mixed precesion storage solution to squeeze a bit more storage out per frame (~16MB saved per frame over using uint16 for the whole grid...)
- batch processing runs X number of simulations back to back (be careful of using to much storage!)
- seed can be set or alternatively allowed to be randomized every run with batch processing incrementing a set seed so its not running the exact same simulation every run of the batch
- frame loading/viewing logic plus rendering done via blender, utilizing the attached .blend file is HIGHLY recommended due to geometry node setup and shader node setup with cycles settings set to be as optimized as possible for the simplicity of the grids
- automated culling systems made to majorly reduce the amount of cells saved while keeping the viewed part as accurate as possible to the true grid so the amount of cells viewed in blender is capped and doesn't break blender (eg. 1M+ cells turning into 1M+ cubes and balloning RAM/VRAM massively)
- documentation on both scripts (generated by claude) 

the script its self has all the configuration inside the script, its a bit spread out but everything is there, will probably add argparse soon but no guarantees.
